# Convert Vite SPA → TanStack Start (SSR, Cloudflare Workers)

**Date:** 2026-06-16
**Status:** Approved
**Branch:** tanstack-start

## Goal

Migrate the existing React 19 + Vite 8 single-page app to TanStack Start with server-side rendering, deployed to Cloudflare Workers, while preserving all existing routes, content, styling, and behavior.

## Current State

- React 19 + Vite 8, Tailwind CSS v4 (`@tailwindcss/vite`), TypeScript 6 strict.
- `react-router-dom` v7 with declarative `<Routes>`/`<Route>` in `src/App.tsx`; pages lazy-loaded via `React.lazy`.
- 10 routes:
  - `/` → Landing
  - `/core-concepts/execution-context-scope` → Week1Index
  - `/core-concepts/execution-context-guide` → ExecutionContextGuide
  - `/core-concepts/execution-context-scope/day-1` … `day-7` → Day1 … Day7
- Router API in use: `BrowserRouter`, `Routes`, `Route`, `Link`, `useLocation` (across `main.tsx`, `App.tsx`, `ContentLayout.tsx`, `DayNav.tsx`, `Landing.tsx`, `Week1Index.tsx`).
- `ThemeProvider` reads `localStorage` + `matchMedia` (already has `typeof window` guards).
- `index.html` owns the document shell + Google Fonts `<link>`.
- Package manager: bun (`bun.lock`).
- `agents.md` convention: run build + lint after every change.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Rendering mode | **SSR** | TanStack Start's purpose; SEO + fast first paint for a content site. |
| Deploy target | **Cloudflare Workers** via Nitro `cloudflare-module` preset | User request; `cloudflare-pages` is deprecated in favor of `cloudflare-module`. |
| Routing | **TanStack file-based routing** | Idiomatic; auto route-tree generation + type-safe `Link`. |
| App directory | **Keep `src/`** | Current canonical TanStack Start convention (`src/routes`, `src/router.tsx`); zero churn for existing components/CSS imports. |
| Build tooling | **Plain Vite** (`tanstackStart()` + `nitro()` plugins) | Current 2026 docs; the older `app.config.ts`/vinxi approach is the RC-era setup and is superseded. |
| Pkg manager | **bun** | Matches existing lockfile. |

## Approach

Idiomatic in-place migration. Keep all components and CSS in `src/` untouched except for router-API swaps. Add a `src/routes/` tree of thin route files that import existing page components. TanStack Start owns the document shell (`__root.tsx`) and SSR entry; `index.html` and the manual lazy/Suspense wiring are removed (per-route code splitting becomes automatic).

## Target Architecture

### Dependencies

- **Add:** `@tanstack/react-router`, `@tanstack/react-start`, `nitro`; dev: `wrangler`, `@tanstack/router-devtools` (optional, dev-only).
- **Keep:** `react`, `react-dom`, `vite`, `@vitejs/plugin-react`, `tailwindcss`, `@tailwindcss/vite`, typescript, eslint stack.
- **Remove:** `react-router-dom`.

### Configuration

- `vite.config.ts` — plugins: `tanstackStart()` first, then `nitro({ preset: 'cloudflare-module', compatibilityDate: '2024-09-23' })`, then `viteReact()` (must come after Start), then `tailwindcss()`.
- `package.json` scripts:
  - `dev`: `vite dev`
  - `build`: `vite build`
  - `start`: `wrangler dev` (local CF runtime); `preview` equivalent.
  - `deploy`: `wrangler deploy`
- `wrangler.jsonc` — Cloudflare Workers config (name, main → `.output/server/index.mjs` or assets binding, compatibility_date `2024-09-23`).
- `tsconfig.json` — add `@tanstack/react-router` module augmentation (register the router for type-safe `Link`/`useRouter`); keep strict flags.

### File map

**New:**
- `src/router.tsx` — exports `getRouter()` → `createRouter({ routeTree, scrollRestoration: true })`, importing `routeTree` from `./routeTree.gen`.
- `src/client.tsx` — `hydrateRoot(document, <StrictMode><StartClient /></StrictMode>)`.
- `src/routes/__root.tsx` — root route: document shell (`<html>`, `<head>` via `HeadContent`, fonts, global CSS via `head().links`, `Outlet`, `Scripts`), `ThemeProvider`, `errorComponent`, `notFoundComponent`, and the no-FOUC theme script.
- `src/routes/index.tsx` → Landing
- `src/routes/core-concepts/execution-context-scope/index.tsx` → Week1Index
- `src/routes/core-concepts/execution-context-scope/day-1.tsx` … `day-7.tsx` → Day1 … Day7 (thin wrappers)
- `src/routes/core-concepts/execution-context-guide.tsx` → ExecutionContextGuide
- `src/routeTree.gen.ts` — auto-generated by the router plugin (not hand-edited).

**Modified:**
- `src/context/ThemeContext.tsx` — SSR-safe initialization; no-FOUC.
- `src/components/ContentLayout.tsx`, `src/components/DayNav.tsx`, `src/pages/Landing.tsx`, `src/pages/core-concepts/Week1Index.tsx` — swap `react-router-dom` imports for `@tanstack/react-router` equivalents.
- `src/pages/core-concepts/navConfig.ts` — type `path` values against registered routes (or targeted cast) for type-safe `Link`.
- `vite.config.ts`, `package.json`, `tsconfig.json` — as above.

**Deleted:**
- `src/main.tsx` (duties move to `client.tsx` + `__root.tsx`)
- `src/App.tsx` (routing moves to file-based routes)
- `index.html` (document shell moves to `__root.tsx`)

### Route file shape (thin wrapper)

```tsx
import { createFileRoute } from '@tanstack/react-router'
import Day1 from '../../pages/core-concepts/Day1'

export const Route = createFileRoute('/core-concepts/execution-context-scope/day-1')({
  component: Day1,
})
```

URLs are preserved exactly — no external link or bookmark breaks.

## Key Implementation Details

### SSR theme handling (no flash-of-wrong-theme)

The theme is persisted in `localStorage` and defaults to dark. To avoid a hydration flash:

1. Server renders with the **dark** theme (deterministic default).
2. A **blocking inline `<script>`** injected in `__root.tsx` `<head>` (rendered before the app) reads `localStorage['js-learn-theme']` and applies the `light` class to `document.body` before first paint.
3. `ThemeProvider` initializes client-side from the class already on `document.body` (set by the script), so React hydration matches what was painted. The existing `useEffect` continues to persist changes.

### Type-safe `Link` with dynamic paths

`navConfig.ts` and `getDayNav()` produce path strings dynamically. TanStack Router's `<Link to>` is strictly typed to known route ids. Handling:
- Define `navConfig` path values as the registered route paths and cast via the router's registered type, or use a minimal `as` cast at the `Link` call sites (`to={item.path as any}` is avoided; prefer typing `NavItem.path` against `FileRoutes['to']`).
- `useLocation()` from `@tanstack/react-router` returns `{ pathname }` — drop-in replacement for the current `location.pathname` usage in `ContentLayout`.

### Code splitting

Remove manual `React.lazy` + `Suspense`. TanStack Router splits per route file automatically. The current `AppLoading` fallback is no longer needed; a route-level `pendingComponent` can be added later if desired (YAGNI for now).

## Verification

Per `agents.md` (build + lint after changes):

1. `bun run dev` → Vite dev server starts; all 10 routes respond with SSR HTML (view-source shows content, not just `<div id="root">`).
2. Client navigation works; active sidebar link highlights via `useLocation`/`activeProps`.
3. Theme toggle persists across reloads with no flash.
4. `bun run build` → typecheck (`tsc`) passes, `vite build` produces `.output/` for the Cloudflare module worker.
5. `bun run lint` → ESLint passes.
6. (Optional) `wrangler deploy` deploys to Cloudflare Workers.

## Risks & Mitigations

- **`Link` type friction with dynamic paths** → handled via typed nav config / targeted casts (see above).
- **Tailwind v4 + `tanstackStart()` plugin interaction** → Tailwind v4's `@tailwindcss/vite` plugin composes normally inside the `vite.plugins` array; global CSS imported via `?url` and referenced in `head().links`.
- **Cloudflare Workers Node-API limits** → app is pure static content with no server-only deps, so worker compatibility is straightforward; SSR uses the Nitro edge preset.
- **Generated route tree drift** → never hand-edit `routeTree.gen.ts`; it regenerates on dev/build.

## Out of Scope (YAGNI)

- Server functions, API routes, middleware, loaders, `defer()`/streaming — not needed for static content.
- Data fetching, auth, DB — none exists.
- Path alias migration beyond optional `~/` (relative imports retained to minimize diff).
- Redesigning UI/UX — the migration is behavior-preserving.
